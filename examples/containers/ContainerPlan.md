# Plan: Fix theories/ compilation after containers v0.7 update

## Context

The `lib/` modules were regenerated from containers v0.7 and compile with Coq 8.20. However, the `theories/` proofs depend on functions that were **skipped** during the v0.7 update:

- `fromDistinctAscList`, `fromDistinctDescList`, `fromAscList`, `fromDescList` (Set and Map)
- `fromAscListWithKey`, `fromDescListWithKey`, `fromAscListWith`, `fromDescListWith` (Map only)

The proofs in `SetProofs.v` and `MapProofs/FromListProofs.v` deeply decompose the old v0.6 implementations (which used `deferredFix` with `create`/`go` local functions). The v0.7 Haskell rewrote these functions entirely (stack-based algorithm), so we can't just un-skip them — we need to **restore the old v0.6 Coq definitions** via `redefine` edits so the proofs remain valid.

Additionally, IntMap/IntSet `link` now delegates to a new `linkWithMask` helper, which changes the unfolding behavior and may break `IntMapProofs.v`/`IntSetProofs.v`.

The user's constraint: **do not admit or remove preexisting theorems**.

## Step 2: Handle IntMap/IntSet `link` → `linkWithMask` change

In v0.7, `link` for IntSet and IntMap was refactored to call `linkWithMask`:
```coq
(* New v0.7: *)
link p1 t1 p2 t2 = linkWithMask (branchMask p1 p2) p1 t1 t2
(* Old v0.6: *)
link p1 t1 p2 t2 = let m := branchMask p1 p2 in
  if zero p1 m then Bin (mask p1 m) m t1 t2 else Bin (mask p1 m) m t2 t1
```

`IntMapProofs.v` (line 1195) unfolds `link`. With the new definition, unfolding `link` gives `linkWithMask (branchMask ...) ...` instead of the direct if-then-else. This will break proof scripts.

**Approach**: Check `IntMapProofs.v` and `IntSetProofs.v` for `link` unfolding. Two options:
- **Option A (preferred)**: Add `unfold linkWithMask` after every `unfold link` in the proof files. This is minimal and preserves all theorems.
- **Option B (fallback)**: `redefine` IntMap/IntSet `link` to the old direct implementation. This avoids touching proof files but diverges from the generated code.

Try Option A first during implementation, fall back to Option B if too many proof sites.
